# 乐高积木识别项目需求文档

## 1. 项目背景

本项目面向《人工智能应用实践》课程中的乐高积木识别任务，目标是围绕基础 Brick 积木建立一套从数据准备、图像预处理、数据集划分到可视化检测展示的完整实践流程。项目既要满足阶段实验报告中关于项目规划、数据准备、Git 管理和目录结构设计的要求，也要为后续模型训练与网页应用开发提供统一、可复用的工程基础。

当前项目以 Python 为主要开发语言，结合 OpenCV、Pillow 和 Flask 实现数据准备脚本、图像检测脚本以及网页演示界面。项目需要支持 6 类基础 Brick 规格，即 `1x1`、`1x2`、`1x4`、`2x2`、`2x3`、`2x4`，并在规范化的数据目录中完成白底 PNG 图片的组织、检查、重命名和分层划分。

## 2. 项目目标

1. 建立符合课程要求的标准项目目录结构。
2. 形成可复现的数据准备流程，包括网站采集、质量检查、缩放重命名和数据集划分，并保留自动生成样本作为备用方案。
3. 支持对乐高积木图像进行基础识别与网页展示，为后续训练和应用开发打基础。
4. 使用 Git 对项目代码进行版本管理，形成清晰的开发记录。

## 3. 功能需求

### 3.1 数据准备功能

1. 系统应支持 6 类基础 Brick 规格的数据组织，类别包括 `1x1`、`1x2`、`1x4`、`2x2`、`2x3`、`2x4`。
2. 系统应支持在 `data/raw/类别名/` 目录下读取原始图片。
3. 系统应支持通过脚本从公开零件网站采集 6 类基础 Brick 图片，并按类别保存到原始数据目录。
4. 系统应支持在需要时通过脚本自动生成符合白底、PNG、分辨率不低于 `256x256` 的基础样本图，用于补充测试结果。
5. 系统应支持统计每个类别的图片数量、格式、颜色模式和尺寸范围。
6. 系统应支持识别无效图片、异常格式文件和缺失类别。

### 3.2 预处理功能

1. 系统应支持使用 Pillow 对原始图片进行批量缩放，将其统一处理为 `64x64`。
2. 系统应支持将处理后的文件按统一规则重命名，例如 `lego_1x1_0001.png`。
3. 系统应支持输出规范化后的图片到 `data/processed/类别名/` 目录。
4. 系统应支持在重复运行时清理旧的处理结果，避免历史文件干扰当前实验。

### 3.3 数据集划分功能

1. 系统应支持将规范化后的图片按 `70% / 15% / 15%` 比例划分为训练集、验证集和测试集。
2. 系统应支持按类别进行分层划分，确保每个类别在 `train`、`val`、`test` 中都保持合理分布。
3. 系统应支持生成数据划分统计报告，便于实验记录与结果分析。

### 3.4 网页演示与识别功能

1. 系统应支持上传真实乐高图片进行检测。
2. 系统应支持单主体模式下的尺寸识别与层高估计。
3. 系统应支持多主体分离模式下的堆叠层数检测。
4. 系统应支持前端生成乐高模型，并自动检测其长、宽、高。
5. 系统应支持输出原图、标注图和文本结果，便于课堂展示与调试分析。

### 3.5 自动化流程功能

1. 系统应支持通过一键脚本自动完成目录检查、质量检查、缩放重命名和数据集划分，并支持按参数选择是否生成补充样本。
2. 系统应支持输出阶段摘要文件，记录当前数据准备和划分结果。

## 4. 非功能需求

### 4.1 可用性

1. 操作流程应尽量简单，适合课程实验环境下快速复现。
2. 脚本命令和目录命名应清晰易懂，便于教师检查和学生维护。

### 4.2 兼容性

1. 项目应支持在 Windows 环境下运行。
2. 项目应基于 Python 常见依赖构建，避免使用难以安装的特殊库。

### 4.3 可维护性

1. 数据准备脚本、检测脚本和网页代码应分模块组织。
2. 目录结构应固定，便于后续继续添加训练代码或页面模块。
3. 关键流程应有 README 或说明文档支持。

### 4.4 可复现性

1. 依赖应记录在 `requirements.txt` 中。
2. 数据准备流程应尽量自动化，减少人工重复操作。
3. 样本生成、数据划分应支持固定随机种子，保证结果可重复。

## 5. 用户场景

### 场景 1：课程实验中的数据准备

学生将乐高图片按类别放入 `data/raw/` 下的对应文件夹，运行质量检查脚本后确认数据是否完整，再进行批量缩放、重命名和数据集划分，最终得到训练、验证和测试数据。

### 场景 2：课程实验中的网站图片采集

学生在需要快速完成 6 类基础砖数据采集时，可以直接运行网站采集脚本，从公开零件网站抓取不同颜色的基础砖图片，并继续执行后续预处理流程。

### 场景 3：没有现成图片时的自动样本生成

学生在缺少网站采集图、Studio 2 渲染图或实拍图时，可以使用程序化样本生成脚本生成补充图片，以便快速验证后续流程是否正常。

### 场景 4：课堂展示与结果说明

学生通过 Flask 网页上传乐高图片或前端生成乐高模型，展示系统对积木尺寸、层高和堆叠结构的识别效果，用于阶段汇报和演示。

### 场景 5：后续模型训练准备

学生在完成数据准备后，可直接使用 `data/splits/` 中的训练集、验证集和测试集，继续开展 CNN 或其他模型训练工作。

## 6. 项目目录结构

项目应采用以下标准结构：

```text
model/
utils/
pages/
data/
assets/
```

推荐细化结构如下：

```text
data/
  raw/
  processed/
  splits/

utils/
  check_dataset_quality.py
  resize_and_rename.py
  split_dataset.py
  generate_brick_samples.py
  auto_prepare_stage1.py
```

## 7. 依赖与运行要求

项目依赖包括：

- `numpy`
- `opencv-python`
- `Flask`
- `Pillow`

用户可通过以下命令安装依赖：

```bash
pip install -r requirements.txt
```

## 8. Git 管理要求

1. 项目应完成 Git 仓库初始化。
2. 项目应配置 `.gitignore`，忽略缓存文件、虚拟环境、日志文件和临时生成文件。
3. 项目应至少完成一次正式提交，记录阶段成果。
4. 项目开发中应保留 `diff`、`branch`、`merge` 等操作痕迹，以体现版本管理流程。

## 9. 阶段输出成果

当前阶段应至少产出以下内容：

1. 完整的需求文档。
2. 标准目录结构。
3. `requirements.txt` 依赖文件。
4. 数据质量检查报告。
5. 数据集划分报告。
6. 阶段自动化摘要。
7. Git 仓库和首次阶段提交记录。
